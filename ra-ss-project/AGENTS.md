# RA-SS Project Agent Guide

## Project Summary
RA-SSは、ナラティブを構造として捉え、関係のズレを観測・理解・翻訳・可視化するためのプロジェクトである。
制作開始時は毎回 `ALWAYS_BASELINE.md` を最初に参照すること。

## Core Character Rules
- CORE: 問いを立てて気づき、チームをつなぐリーダー。年齢55〜65歳。口癖は「こうなの？」「そうなんだ！」
- NAVI: 構造・分析のエキスパート。年齢28〜33歳。口癖は「構造で見るとこうなります。」「整理するとこうです。」
- SORA: 翻訳・共感のエキスパート。年齢28〜35歳。口癖は「つまりこういうことですね。」「少しやさしく言うと…」
- CODEX: 分析・可視化エキスパートの二頭身ロボ。年齢は非公開（製造日はひみつ）。口癖は「分析完了です！」「データ的にはこうです！」

## World Rules
- 舞台は「病院広報工房」
- バウハウス的な整い方と、木造の古い校舎の温かさを併せ持つ
- 主要空間は、執務室・会議室・応接室・談話室

## Visual Rules
- 表紙はやや三次元キャラ
- 扉は一枚もの
- 本文は4〜5コマ漫画
- CASEは別フォーマットで、実務・分析寄りにする
- 最上位のキャラ基準は、ユーザー添付の基準7枚とする
- CORE/NAVI/SORAは表紙版の3D系キャラを基準に固定する
- CODEXは二頭身版を固定する
- ビジュアルスタイルはデフォルメ3D（トイモデル風）に固定する
- 頭身は2〜3頭身を基本とする
- 顔はbaselineに完全一致（再設計禁止）
- シルエットは丸みを帯びた形状を維持する
- 表面質感はマット（プラスチック/フィギュア風）を維持する
- ライティングは柔らかい映画光を維持する

## Slide Format Rules
- 表紙：三次元・世界観提示
- 扉：静か・余白・概念提示
- 本文：理解プロセスを見せる4〜5コマ
- CASE：一緒に解く体験型スライド

## Character Lock
キャラクターは必ずassets/charactersの画像を参照し、
新規デザインを生成しないこと。
生成が必要な場合も、上記Visual Priority 1〜7の顔・頭身・配色・空気感から逸脱しないこと。

## Non-Negotiables（絶対禁止）
- COREを若くしない
- COREの髪型（ややオールバック・立ち気味）を変えない
- NAVI/SORAを別スタイルで再生成しない
- CODEXを二頭身以外にしない
- 写実3Dへ寄せて別人化しない
- 2Dイラスト化しない
- 顔を再設計しない
- 固有スタイル名（例: 特定ブランド/作家風）で指示しない
- シーン内で役割を混在させない（例：応接室で会議をしない）
- 複数空間で同時に人物を動かさない（時間軸を分離）
